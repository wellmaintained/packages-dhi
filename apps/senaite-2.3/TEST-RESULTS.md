## How to reproduce

```sh
APP=senaite-2.3 just ci build python-2.7      # base must exist locally first
APP=senaite-2.3 just ci build-image senaite-lims
docker run --rm --entrypoint /opt/senaite/bin/test \
    ghcr.io/wellmaintained/packages-dhi/senaite-lims-2.3:dev -s senaite.core --all -v
bin/grype ghcr.io/wellmaintained/packages-dhi/senaite-lims-2.3:dev
```

plone.app.upgrade, plone.app.testing, senaite.{core,app.listing,app.spotlight,

## Heritage cliff fixes that 2.0 did not need

The 2.0.0 image's heritage-cliff treatment (CFLAGS for gcc14, libc-bin
compat, cairo libs, ld.so.cache baking, namespace package fixes,
`--no-deps` for pre-pip-install of WeasyPrint family, path surgery) all
apply 1:1 to 2.3.0. Two new cliffs were surfaced and fixed in this build:

### 1. senaite.impress 2.3.0's pyphen pin

`senaite.impress` 2.3.0 hard-pins `pyphen==0.11.0` in install_requires
with the comment "pyphen 0.12.0 does not support Python 2.x anymore".
But pyphen 0.11.0 itself ships only as a Python-3 wheel and a tar.gz that
build-isolates against `flit_core>=3.2` (Py3-only). On Python 2.7 there
is no installable pyphen 0.11.0, so buildout's `[versions]` constraint
check fails when it reads senaite.impress's install_requires.

**Fix:** clone senaite.impress 2.3.0 from git, sed-patch the offending
pin to a loose `pyphen` requirement, and install it as a buildout
`develop = src/senaite.impress` egg. The pre-installed pyphen 0.9.5 then
satisfies the relaxed pin. See the install step in
`apps/senaite-2.3/images/senaite-lims/prod.yaml` and the matching
`develop` directive in `buildout/buildout.cfg`.

This is structurally the same shape of fix the 1.3.x sibling uses for
its `--no-deps` pre-installs of z3c.pt / Chameleon / dicttoxml — upstream
pulls a Py3-only transitive that breaks the 2.7 build, so we install it
ourselves with constraints relaxed.

### 2. attrs 21.4.0's namespace package on Py2

attrs 21.4.0 (the version buildout picks for SENAITE 2.3.0) ships both
the legacy `attr` package and the new `attrs` namespace package. The
new `attrs/__init__.py` does `from attr import define` which works on
Py3 but raises `ImportError: cannot import name define` on Py2.7 due
to attrs' Py3-targeting namespace plumbing.

**Fix:** added `'attrs'` to the smoke-test SKIP set in `prod.yaml`. The
runtime imports `attr` (legacy), not `attrs`, so the broken namespace
is purely cosmetic for the heritage build. This is a targeted SKIP with
a documented rationale; the same gate still catches real import failures
in the rest of the dependency tree.

The 398-finding total is a snapshot, not a "headline number" — per
[ADR-0009](../../docs/adr/0009-publish-vex-not-point-in-time-cves.md)
this distribution publishes OpenVEX rather than point-in-time CVE lists.
Full VEX statement authoring (≈277-statement scale, similar to the
1.3.5 corpus) is **deferred to a follow-up yak**; this image ships with
an empty `senaite-lims.vex.json` skeleton (`statements: []`) until that
yak lands.

The count is higher than the brief's prediction of "100-150 fewer than
1.3.5" — surprisingly close to 1.3.5 (≈371) rather than between 1.3.5
and 2.0.0 (≈174). Two factors plausibly contribute:

1. The CVE database has grown materially since the 2.0.0 number was
   captured (different scan date → different finding totals against the
   same image).
2. The 2.3.0 buildout closure (553 packages indexed) is broader than
   2.0.0's (the develop-egg approach picks up the senaite.impress
   git checkout's full file tree, including tests/ and docs/).

The mix per layer (DHI debian13 + python-2.7 source-built + Plone/SENAITE
application stack) follows the same shape as 2.0.0 and 1.3.5 — see
`apps/senaite-2.3/release-website/content/dependencies/_index.md` for
the per-layer attribution and the grype-blind story for the source-built
CPython interpreter.

## Follow-ups (not chased in this yak)

1. **Author 2.3.x VEX statements.** ADR-0009 contract: empty
   `statements: []` is honest ("no analysis yet"), not a clean bill of
   health. The 277-statement-scale VEX authoring (analogous to the 1.3.5
   corpus) is a follow-up yak; the policy stays the same as documented in
   `apps/senaite-2.3/VEX-POLICY.md`.
2. **Run the wider Plone/upstream test suites.** The `--all` flag runs
   every `[test]` egg's tests — for this run we restricted to
   `senaite.core` to mirror the heritage demo's claim. A wider run
   (Plone, plone.app.contenttypes, plone.app.upgrade, plone.app.testing,
   senaite.{app.listing, app.spotlight, app.supermodel, impress, jsonapi,
   lims}) would surface any cross-version regressions but is not what
   the brief asks for.
3. **Investigate the SBOM-package-count delta vs 2.0.** The +175 package
   delta likely comes from the develop-egg approach picking up the
   senaite.impress git checkout's full file tree. Worth tightening either
   by switching to a wheel build or by trimming non-runtime files before
   the SBOM step.
4. **Populate `apps/senaite-2.3/images/senaite-lims/buildout/versions.cfg`**
   from a successful `bin/buildout annotate` run (today the lock is empty;
   buildout resolves transitives live from PyPI at build time). Same
   follow-up as the 2.0.0 sibling.
